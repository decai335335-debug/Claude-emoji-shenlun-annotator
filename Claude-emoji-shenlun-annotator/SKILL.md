# 申论深度标注 Skill

## 一、适用场景

- 用户给出**申论原题（题干 + 给定资料）**，要求生成"标注版 Markdown"。
- 用户要求按照《申论刷题第二期》20 天范本的格式继续标注新材料。
- 用户要求从已有的非标注文本中提取采分点并按统一样式标注。

## 一·五、执行前置检查（**强制，先于任何标注动作**）

在动笔之前，**必须**判定输入是"**原料**"还是"**成品**"：

1. **文件名检查**：若文件名含 `标注版`、`annotated`、`_emoji_`、`_v2` 等成品标识 → 高风险。
2. **内容抽样检查**（扫描前 30 行即可判定）：
   - 若正文中已出现 `` `【{emoji}...】` `` 反引号标签，
   - 或已含 `## 📊 ...标注汇总` 表格，
   - 或已有大量 `**...**` 加粗，
   → **判定为已标注成品**。
3. **判定动作**：
   - ✅ 原始材料（无标签、无加粗、无汇总表）→ 正常执行第七节标注流程。
   - ⚠️ 已标注成品 → **停下来**，不要自动"再标一遍"。向用户明确反馈：
     > "该文件已是标注版，本 skill 需要**未标注的原始给定资料**作为输入。
     > 请确认以下选项之一：
     > (A) 提供原始真题给定资料纯文本；
     > (B) 我帮你**剥离**现有标签/加粗得到原料后重标（会丢失原标注信息）；
     > (C) 切换为 **review 模式**——仅校对和优化现有标注，不新增要点；
     > (D) 对现有汇总表做 **answer-extraction**——直接基于标签生成答题要点稿。"
   - 让用户选择后再继续。**严禁默认覆盖重写已标注文件** —— 那只是换皮，不是挖掘新采分点。

**原则**：skill 是"从原料炼出采分点"，不是"把成品换个标签重排"。拿成品当原料 = 任务逻辑闭环失败。

## 二、文件整体结构（严格遵守）

输出必须为一份完整 Markdown，模板如下：

```markdown
# 📝 第X天：{考试类型} - {主题}

**题干**：{原题干文字}。({分值}分)

---

## 【给定资料标注版】

{段落1（已标注）}

---

{段落2（已标注）}

---

... 其他段落 ...

---

## 📊 {主题}标注汇总

| {维度} | {内容} | {数据/案例} |
|:---|:---|:---|
| **xxx** | xxx | xxx |

**核心经验**：xxx + xxx + xxx
```

关键点：
- 顶部 `# 📝` 一级标题。
- 题干用 `**题干**：` 前缀，末尾保留分值。
- 每个自然段用 `---` 分隔（前后空行）。
- 末尾**必须**生成 `## 📊 ...标注汇总` 表格 + "核心经验 / 核心启示 / 核心机制"一句话总结。

## 三、行内标注格式（最核心的规则）

### 1. 加粗

用 `**text**` 包裹**原材料术语、数字、机构名、政策名、人名+身份、核心动词短语**。
- ✅ `**农村农业局**派了**技术员点对点指导**`
- ✅ `**500多万株**`、`**40多万元**`
- ❌ 不要加粗长句、形容词、连接词。

### 2. 行内标注（括号 + emoji + 冒号）

**格式**：`` `【{emoji}{中/英标签}: {精炼内容}】` ``

- 整体包在**反引号**里（`` ` ` ``），保证在渲染时与正文区分。
- emoji **紧贴**标签词，之间不加空格。
- 冒号**后**加一个半角或全角空格均可，保持整篇一致。
- 标注**紧跟**被标注句末，**不换行**。
- 多个标注**连写**，不加分隔符：`` `【...】【...】【...】` ``

示例：

```
"市**农村农业局**派了**技术员点对点指导**我们。`【🔧Assistance measure 1: 技术下乡——农村农业局派技术员点对点指导】【💯Data: 500多万株、40万纯利润】`
```

### 3. 子项表达

标签内部若需展开多点，用 `——` 引出，用 `+` 或 `、` 连接：
- `【🔧Approach: 信用评价扩围+M级企业纳入+以信换贷】`
- `【🔧Measures: 改善微循环+贯通主动脉】`

### 4. 语言风格

**中英混标均可**，沿用范本习惯（20 篇中以英文标签 + 中文内容为主）：
- `【⭐Core viewpoint: ...】` / `【⭐核心观点：...】`
- `【🔧Approach 1: ...】` / `【🔧做法1：...】`

在同一份文件内**保持统一**；续写他人已有文件时沿用其风格。

## 四、Emoji 语义体系（记住 8 大高频标签）

| 语义 | emoji | 典型标签 | 用途 |
|:---|:---:|:---|:---|
| **核心观点** | ⭐ 💎 🏆 ✨ | Core viewpoint / Core experience / Core concept | 全段主旨、结论升华、点题句 |
| **措施做法** | 🔧 ✅ 🛠️ ⚙️ | Approach / Measure / Solution / Mechanism | 政府/企业具体做法，最常用 |
| **问题困境** | ⚠️ ❗ 🚨 | Problem / Challenge / Core problem | 负面现象、痛点、瓶颈 |
| **数据事实** | 💯 📊 📈 | Data / Key data / Achievement data | 数字、比例、规模、金额 |
| **成效反馈** | 🚀 🎉 | Effect / Achievement / Result | 做法之后的积极变化 |
| **背景政策** | 🌐 🏛️ 📍 | Background / Policy background | 非采分的时空/政策铺垫 |
| **概念定义** | 📖 💡 🧩 | Concept / Definition / Idea | 新名词、新提法解释 |
| **目标愿景** | 🏆 🎯 | Goal / Vision / Direction | "打造……""建设……""迈向……" |

次高频:
- `👥` 群众声音 / `💬` 原文引用 / `🔎` 调研分析 / `🏷️` 分类标签 / `💎` 亮点价值。

**一句话原则**：**做法→🔧、问题→⚠️、数据→💯、背景→🌐、结论/观点→⭐、成效→🚀、概念→📖**。

### 四·五、**丰富度要求（防止单调）** ⚠️ 重点

仅靠 `⭐/🔧/⚠️/🌐` 四个裸标签会让整篇标注看起来**非常单调**。要学范本（如《政府企业合力治水》）,给每个 emoji **挂"角色/阶段/维度限定词"**,形成标签**变奏**。

**必须做到**：每段若出现同类 emoji ≥ 2 次,第二次起**必须换限定词**,不得简单重复。

#### 常用"二级限定词"词库（按 emoji 分组,直接抄用）

- **🌐 背景类**：`Background` / `Policy background` / `Policy context` / `Tech background` / `Historical background` / `Scene` / `Industry background`
- **⭐ 观点类**：`Core viewpoint` / `Core experience` / `Core concept` / `Core achievement` / `Core conclusion` / `Core method` / `Core strategy` / `Core value`
- **🔧 做法类（按主体拆）**：
  - 政府侧：`Government action` / `Government service` / `Policy tools` / `Regulatory innovation`
  - 企业侧：`Enterprise action` / `Tech innovation` / `Work method` / `System architecture` / `Service innovation`
  - 协同侧：`Cooperation mechanism` / `Cross-dept coordination` / `Diagnosis method` / `Construction model`
- **⚠️ 问题类**：`Problem` / `Core problem` / `Core problem diagnosis` / `Challenge` / `Pain point` / `Risk`
- **💯 数据类**：`Data` / `Key data` / `Time span` / `Milestone` / `Contrast data` / `Scale data` / `Achievement data`
- **🚀 成效类**：`Effect` / `Achievement` / `Function effect` / `Contrast effect` / `Achievement promotion` / `Social effect` / `Economic effect`
- **💡 理念类**：`Idea` / `Concept` / `Enterprise motivation` / `Enterprise thinking` / `Enterprise opportunity` / `Core concept` / `Key insight` / `Principle`
- **📖 概念类**：`Concept` / `Definition` / `Term` / `New concept`
- **🏆 目标类**：`Goal` / `Vision` / `Direction` / `Strategic positioning`

#### 挂限定词的 3 种视角（帮助你选词）

1. **主体视角**：政府 / 企业 / 群众 / 专家 / 第三方 → `Government action` vs `Enterprise action` vs `Public feedback`。
2. **阶段视角**：发现问题 → 研讨 → 决策 → 执行 → 验收 → 推广 → `Enterprise thinking` → `Diagnosis method` → `Core experience` → `Achievement` → `Achievement promotion`。
3. **维度视角**：技术 / 制度 / 资金 / 人才 / 协同 → `Tech innovation` / `System architecture` / `Cooperation mechanism` / `Policy tools`。

#### 反例 → 正例

❌ 单调版（你当前的风格）：
```
【🌐Background: ...】...【🌐Background: ...】...【⚠️Problem 1: ...】...【⚠️Problem 2: ...】...【⚠️Problem 2-1: ...】
```

✅ 变奏版（目标风格,学治水范本）：
```
【🌐Policy background: "五水共治"方针】【🔧Government action: 发布公告进行污染源排查】【💡Enterprise opportunity: 主动请缨，利用技术优势】
【⭐Core experience: 政府购买服务，企业补充政府力量】【💯Key data: 32条河道】
【🔧Work method: 实地蹲守、逐一排查】【💯Key data: 30多人、300多天、1万多个污染源】【🚀Achievement: 系统摸清污染源情况】
【🔧Innovation: 主动研发城市排涝指挥控制系统】【🚀Effect: 实时掌握+预警推送+快速反应】【💯Key data: 两个月研发周期】
```

每个标签**换一个限定词**,标注层次立刻丰富。**永远不要连续两次用同一个裸"Background/Problem/Measure"** —— 必须换角度。

## 五、标注内容的选取标准

对每一段材料，按以下顺序识别要点：

1. **领导讲话、总结句、主旨句** → `⭐核心观点`
2. **具体动词短语（派、建、推行、打造、出台、联合、搭建）** → `🔧措施/做法`
3. **数字、百分比、金额、时间跨度** → `💯数据`
4. **"问题""瓶颈""短板""困难""……不足""……难"** → `⚠️问题`
5. **"有效……""提升……""实现……""解决了……"** → `🚀成效`
6. **政策文件、会议、历史阶段、地域背景** → `🌐背景`
7. **新概念引号内术语（"数字孪生""东数西算""柔的方法"）** → `📖概念`
8. **目标愿景（"打造……圈""建设……体系"）** → `🏆目标`

一句话多要点时 **连续堆标签**，不要合并。

## 六、末尾汇总表生成规则

1. 表格标题固定 `## 📊 {主题词}标注汇总` 或 `## 📊 ...核心标注`。
2. **列数 2–4 列**，首列用 `**加粗**` 类别名。
3. 对齐语法：`|:---|:---|` (左对齐) 或 `|:---:|` (居中，序号列)。
4. 表格之后补**一句话核心经验**，用 `+` 连接要素：
   - `**核心经验**：居民参与 + 多方共担 + 市场化运营`
   - `**核心启示**：守 = 保留特色；新 = 与时俱进`
   - `**核心机制**：政府牵头 + 企业参与 + 群众监督`

## 七、标注工作流（你的执行步骤）

0. **前置检查**（见第一·五节）：确认输入是原料而非成品；若是成品，停下询问用户。
1. **读题干** → 判定文体（概括题/对策题/分析题/应用文/大作文），题干决定要点颗粒度。
2. **分段处理**，每段：
   - (a) 通读识别主体、动作、数据、问题、成效；
   - (b) **加粗**原文术语；
   - (c) 在相应句末**追加**`` `【...】` ``标签，多标签连写；
   - (d) 段末用 `---` 分隔。
3. **不要改写正文**，只做"加粗 + 追加标签"两件事。引号内原话完整保留。
4. 全文处理完毕后 → 生成末尾汇总表 + 一句话核心经验。
5. **写入 Markdown 文件**：按第十一节《输出目录规范》确定路径（utf-8，保留原换行风格）。
6. **同步生成 PDF**（本 skill 默认启用，除非用户明确说"只要 md"）：
   - 目标路径：按第十一节规则,置于 md 所在"标注版"目录下的 **`pdf` 子目录**,文件名与 md **同名**,扩展名 `.pdf`。
   - 例：`.../标注版/标注版_XXX.md` → `.../标注版/pdf/标注版_XXX.pdf`。
   - 调用方式：优先 `Skill(anthropic-skills:convert-md-to-pdf)`；不可用时回退到 `pandoc` / `markdown-pdf` / `mdpdf` 等本地命令；仍不可用则报错提示用户安装。
   - PDF 必须支持中文(含 emoji) —— 若默认字体渲染异常，改用 `Microsoft YaHei` / `Noto Sans CJK SC` / `SimSun` 等 CJK 字体。
   - 完成后向用户同时返回 md 路径和 pdf 路径两条点击链接。
7. 交付清单:md 文件路径 + pdf 文件路径 + 标注统计(⭐🔧⚠️💯🚀📖 各几条)。

## 八、常见错误与禁忌

- ❌ 把标签写成独立一行 → 必须紧跟句末。
- ❌ 忘记反引号包裹 → `` `【...】` `` 是硬性要求。
- ❌ 在标签里写长句解释 → 标签要**短语化、术语化**，≤ 20 字为宜。
- ❌ 整段堆一个超长标签 → 应拆分为 2–4 个语义独立的标签。
- ❌ 改写、压缩、翻译原文 → 只可加粗与追加。
- ❌ emoji 用错语义（用 🔧 标问题、用 ⚠️ 标数据）→ 参照第四节对照表。
- ❌ 中英风格在同一文件里混乱切换。
- ❌ 漏掉末尾 📊 汇总表。
- ❌ **标签单调** —— 全篇只用 `⭐/🔧/⚠️/🌐` 四个裸词。必须按第四·五节挂"二级限定词",每段同 emoji 重复出现时**换一个限定词**（如第一个 🌐 用 `Policy background`、第二个用 `Tech background`）。自检:读完一篇,限定词种类应 ≥ 12 种,否则判定为单调,重做。

```
问题句：…… **瓶颈** ……`【⚠️Problem N: 核心痛点】`
```

## 十·五、分段写入规范（防卡死，硬性约束）

大文件一次性 `Write` 整篇标注 → 容易触发输出 token 超限/沙箱超时,表现为"卡死"。**必须**按以下策略执行:

### 写入流程

1. **第一次落盘**:只写**文件骨架**到最终路径(第十一节定义),内容为:
   - 顶部 `# 📝 ...`标题
   - `**题干**：...`
   - `## 【给定资料标注版】` 章节头
   - `<!-- WIP: 正在分段追加,共 N 段 -->` 占位注释
2. **逐段追加**:用 `Edit` 把 `<!-- WIP -->` 替换为"第 1 段 + 新的 WIP 标记",再下一次替换 WIP 为"第 2 段 + WIP",依次推进。**每次只处理 1 段**(≤ 一个自然段原文 + 对应标注)。
3. **末段处理完毕后**,再 `Edit` 把最后的 WIP 替换为 `## 📊 ...标注汇总` 表 + 核心经验总结。
4. **md 全部落盘完成后**,**单独一轮**调用 PDF 转换,不与 md 写入并行。

### 禁令

- ❌ **禁止**用单个 `Write` 写入超过 **2000 字**的最终内容 —— 必须拆成多次 `Edit`。
- ❌ **禁止**"先写临时文件,再脚本搬运"的绕路 —— 直接写到目标路径,一步到位。
- ❌ **禁止**在同一轮 tool_use 里并行:mkdir + write + pdf convert。按顺序来。
- ❌ **禁止**一次 tool_use 调用 3 个以上写入类工具(Write/Edit)。

### 断点续写

若用户说"接着写"/"继续"/"从第 N 段开始"/"别重做":
- 先 `Read` 当前文件末尾 30 行,定位 `<!-- WIP -->` 或最后一段,
- **不要重写前文**,只追加剩余段落。
- 明确告诉用户当前进度("已完成 3/7 段,继续第 4 段")。

### 预览模式（用户可触发）

用户说"先预览 3 段""别落盘""只看效果"时:
- 直接在回复里输出 3 段样例(不调用 Write),
- 等用户确认风格 OK 后再按上述分段流程落盘。

## 十一、输出目录规范（路径硬性约定）

**给定输入 md 路径**：`<原目录>/<原文件名>.md`
例：`C:\Users\15403\Documents\天琦省考\山东省考申论系统班\XXX.md`

**自动派生输出路径**：

| 产物 | 目标路径 | 示例 |
|:---|:---|:---|
| 标注版 md | `<原目录>/标注版/标注版_<原文件名>.md` | `...\山东省考申论系统班\标注版\标注版_XXX.md` |
| 标注版 pdf | `<原目录>/标注版/pdf/标注版_<原文件名>.pdf` | `...\山东省考申论系统班\标注版\pdf\标注版_XXX.pdf` |

**规则**:
1. **原目录** = 输入 md 所在的目录（`dirname`）。
2. 在原目录下**若无 `标注版` 文件夹 → 自动 `mkdir -p` 新建**。
3. 在 `标注版` 文件夹下**若无 `pdf` 子文件夹 → 自动新建**。
4. md 文件名加 **`标注版_`** 前缀（若输入文件名已含 `标注版` 则**不重复加**）。
5. pdf 文件名**与 md 同名**,仅目录不同（md 在 `标注版/`,pdf 在 `标注版/pdf/`）。
6. **绝不**写入输入 md 所在的原目录根（避免污染源文件),更**绝不覆盖输入源 md**。
7. 若输入 md 本身已位于 `标注版` 目录内（说明是二次加工）→ 询问用户意图,不自动层层嵌套。

**伪代码**:
```
input_md     = C:\...\山东省考申论系统班\XXX.md
base_dir     = dirname(input_md)            # C:\...\山东省考申论系统班
basename     = stem(input_md)               # XXX
annot_dir    = base_dir / "标注版"          # mkdir -p
pdf_dir      = annot_dir / "pdf"            # mkdir -p
out_md       = annot_dir / f"标注版_{basename}.md"
out_pdf      = pdf_dir   / f"标注版_{basename}.pdf"
```

**交付回执格式**(一律用绝对路径):
```
✅ 标注版 md : <out_md>
✅ 标注版 pdf: <out_pdf>
```

## 十、PDF 导出实操细则

- **触发时机**：md 写入成功之后**立即**执行,默认开启,无需询问。
- **命名规则**：严格同名,`{basename}.md` → `{basename}.pdf`,不要加后缀、不要改目录。
- **首选方案**:调用已安装的 `anthropic-skills:convert-md-to-pdf` skill,传入 md 路径即可。
- **回退方案**(按优先级尝试):
  1. `pandoc "xxx.md" -o "xxx.pdf" --pdf-engine=xelatex -V mainfont="Microsoft YaHei"`
  2. `npx md-to-pdf "xxx.md"` (Node 环境)
  3. `mdpdf "xxx.md" "xxx.pdf"`
- **中文/emoji 故障排查**:若 PDF 中中文或 emoji 显示为方框 → 更换 CJK 字体 + emoji fallback 字体(如 `Segoe UI Emoji`、`Noto Color Emoji`)重试一次。
- **失败处理**:导出失败不要静默吞掉,在最终回复里说明失败原因 + 建议命令,md 本体仍视为交付成功。
- **用户可关闭**:若用户显式说"只要 md 不要 pdf"/"跳过 pdf",则省略步骤 6。
